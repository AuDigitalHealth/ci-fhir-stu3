SD.14 StructureDefinition-encounter-es-1

Summary of an Encounter for an Event [Draft]

Implementation guidance

For the overarching usage scenarios in this implementation guide it is expected that:

  • status will be ‘finished’
  • the following elements are not likely to be sent but are encouraged:
    • class
    • reason
  • type will support categorisation of the event from which the medicines list was generated, for example a code indicating a home medicines review
  • period is equal to, or prior to, Composition date and List date

For the expected usage scenario of a PSML document, when sending type, the codes from Medicines Review Type are preferred.

SD.14.1 Formal Views of Profile Content

The official URL for this profile is: 

http://ns.electronichealth.net.au/ci/fhir/3.0/StructureDefinition/encounter-es-1

The purpose of this profile is to provide a summary statement of an encounter for an event between a patient and a healthcare provider. This can include a narrative description of the encounter that is not captured in other fields and include a summary of the issues or problems, management strategies, outcomes or progress, possible prognosis, and the patient’s understanding of the healthcare event.

Must Support

In the context of this profile Must Support SHALL be interpreted as follows:

  • The system SHALL be able to include the element when authoring a resource where data is available

  • The system SHALL be able to store and retrieve the element

  • The system SHALL be able to process resources containing the element

This profile builds on AUBaseEncounter.
